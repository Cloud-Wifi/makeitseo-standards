# Agent Runtime Compatibility (ARC) v0.1

**Specification:** `agent-runtime-compatibility/v0.1`
**Status:** Draft for v1.0 launch.
**Published:** 11 May 2026 (revised; v0.1 final target ~early November 2026)
**Author:** Daniel Titton, Cloud WiFi Limited
**License:** CC-BY 4.0
**Canonical URL:** https://github.com/Cloud-Wifi/makeitseo-standards/blob/main/agent-runtime-compatibility/v0.1.md

---

## Abstract

Agent runtimes are the architectural layer between AI models and the websites they reach. Following the April 2026 ship dates of Cloudflare Project Think, OpenAI Agents SDK, and Google's framing of Search itself as an "agent manager," the runtime is no longer a component of how AI sees the web — it is the component. The model only sees what the runtime hands it.

This specification defines the vocabulary, taxonomy, and methodology for measuring whether a website is *compatible* with a given agent runtime. It introduces the term "runtime readability" as the property a website must have to be legible to a runtime.

**ARC is a measurement vocabulary spec, not a pillar spec.** It defines runtime taxonomy, characteristic vocabulary, applicability matrix structure, and per-(check × runtime) finding emission. It is referenced by the three outcome pillar specs in the MakeitSEO Standards Programme — `seo-scoring/v0.1`, `aeo-scoring/v0.1`, and `geo-scoring/v0.1` — which each contain a runtime-compatibility sub-score grounded in this methodology.

ARC is not a fourth pillar in the score composition. Runtime compatibility is a precondition rather than an outcome, and is measured within the outcome pillar where its failure mode lands. The reasoning is captured in `methodology-adr/v0.1` Decision 2 and Decision 3.

This specification provides:

- A taxonomy of agent runtime categories (browser-execution, headless-fetch, model-native-harness)
- A characteristic vocabulary covering JavaScript execution, session model, sandbox semantics, and emerging properties
- A measurement methodology defining how a website's compatibility with a runtime is assessed
- A customer-site readability framework providing the concrete tests a website operator can run
- A specification for the derived cross-pillar Agent Runtime Readiness view

The specification is published under CC-BY 4.0 to maximise adoption. Implementations are encouraged in any product, framework, or methodology that addresses agent-runtime measurement.

---

## Status

This is the v0.1 draft of ARC. The version reflects two facts: the underlying field is new (the runtime layer became commercially significant in April 2026), and the methodology benefits from real-world implementation feedback before being declared v1.0.

The v0.1 → v1.0 promotion is targeted for 60–90 days after the launch of MakeitSEO's audit platform (~early November 2026). v1.0 incorporates: refinements based on customer audit data, runtime registry expansions absorbing new runtimes that ship between v0.1 and v1.0, and external feedback from any independent implementers.

ARC v0.1 is publishable — sufficient to begin measuring against, sufficient to begin citing, sufficient to begin adopting. It is not complete in the sense that the foundational algorithms of any new measurement domain are not complete on day one.

## Scope

### In scope

- Definition of "agent runtime" as a distinct architectural layer
- Taxonomy of runtime categories observable as of May 2026
- Characteristic vocabulary describing runtime behaviours that affect website compatibility
- Measurement methodology for compatibility assessment
- Customer-site readability framework with concrete tests
- Specification for the derived cross-pillar Agent Runtime Readiness view
- Reference implementation guidance for products and methodologies adopting ARC

### Out of scope

- Specific runtime implementation details proprietary to runtime vendors
- Performance benchmarking of runtimes against each other
- Recommendations on which runtimes a website should optimise for (this is a product-specific decision)
- AI engine citation methodology (covered in `geo-scoring/v0.1` and related specs)
- Search engine ranking methodology (covered in `seo-scoring/v0.1` and related specs)
- Tool-calling protocols such as MCP (Model Context Protocol). MCP defines how an agent invokes structured operations against a service; ARC defines whether a runtime can read a website. These are different layers and ARC does not measure MCP compatibility.

## Definitions

### Agent runtime

An execution environment that mediates between an AI model and external systems (websites, APIs, databases, files). The runtime is responsible for: fetching content, parsing responses, handling authentication, managing session state, executing code (if applicable), and returning results to the model in a form the model can consume.

Examples: Cloudflare Project Think, OpenAI Agents SDK, Anthropic's Agent Skills primitives, Perplexity's fetcher, Gemini's fetcher, Google's Search-as-agent-manager pipeline.

### Runtime readability

The property of a website such that an agent runtime can successfully fetch, parse, and return its content to a downstream model in a form the model can use. A website with high runtime readability is legible to a wide variety of runtimes; a website with low runtime readability is legible only to a narrow subset (or to none).

### Runtime category

A high-level grouping of runtimes by architectural pattern. ARC v0.1 defines three categories: `browser-execution`, `headless-fetch`, `model-native-harness`. The path-versioning convention supports promoting a fourth category in v0.2 if a genuinely distinct pattern surfaces in the months following v0.1 publication.

### Characteristic

A specific behavioural property of a runtime that affects compatibility with websites. Examples: whether the runtime executes JavaScript; whether it persists session state across calls; whether it sandboxes code execution.

### Applicability

The property that a given compatibility check is meaningful for a given runtime. Not every check applies to every runtime. A "session continuity" check is meaningful for a runtime with `session_model = 'durable-execution'` but not for one with `session_model = 'stateless'`. ARC v0.1 introduces the concept of an applicability matrix to express which checks apply to which runtimes.

### Compatibility finding

The result of evaluating a single (check, runtime) pair against a website. A compatibility finding is one of: `compatible`, `incompatible`, `not-applicable`, `indeterminate`.

### Outcome pillar

A measurement pillar in the MakeitSEO Standards Programme that measures a customer outcome (search ranking, answer-surface inclusion, AI citation). The three outcome pillars are SEO, AEO, GEO. ARC findings are routed into these pillars as runtime-compatibility sub-scores rather than producing a fourth pillar score.

### Agent Runtime Readiness

A derived display metric computed across the three outcome pillars' runtime-compatibility sub-scores. Reported alongside pillar scores as a diagnostic. Not itself a pillar score and does not compose into the overall MakeitSEO score.

## Runtime taxonomy

ARC v0.1 defines three runtime categories observable as of May 2026. Categories are not strict — a runtime may exhibit characteristics of more than one category, in which case the dominant category is assigned and secondary characteristics are captured in the runtime's `extensions` metadata.

### browser-execution

The runtime executes a real or browser-equivalent rendering pipeline before handing content to the model. JavaScript executes; the DOM is constructed; client-side rendering is resolved. The model receives content equivalent to what a human user would see in a browser.

**Example as of May 2026:** Cloudflare Project Think (executes JavaScript inside Dynamic Workers sandbox); Googlebot's rendering pipeline as inherited by Google's Search-as-agent-manager surface.

**Implication for websites:** client-side-rendered (CSR) content is visible to browser-execution runtimes. Schema markup injected via JavaScript after page load is visible. JavaScript-driven interactions can be attempted (though not always completed).

### headless-fetch

The runtime issues an HTTP request and parses the raw HTML response. JavaScript does not execute. The DOM is whatever the server delivered. Client-side rendering does not resolve. The model receives the initial HTML payload only.

**Example as of May 2026:** Perplexity's fetcher; Gemini's fetcher; Anthropic's default web fetch.

**Implication for websites:** server-side-rendered (SSR) content is visible. CSR content is invisible. Schema markup must be in the initial HTML payload to be detected. Most lightweight content (text, images, links, metadata) is fully accessible.

### model-native-harness

The runtime is a thin wrapper around an AI model with specific capability primitives (file access, web fetch, code execution as discrete tools the model invokes). The model orchestrates calls; the harness provides safe execution. Web fetch is typically headless within this category, but the harness may expose multiple fetch modes.

**Example as of May 2026:** OpenAI Agents SDK; Anthropic Agent Skills.

**Implication for websites:** similar to headless-fetch in most cases, but the model's orchestration logic affects what's fetched and when. A website should be readable in the same way as for headless-fetch runtimes; secondary properties (predictable URL structures, well-formed structured data) become more important because the model uses them to plan multi-step workflows.

### Fourth-category extension path

A fourth runtime category will be added in v0.2 if a genuinely distinct architectural pattern surfaces during the months following v0.1 publication. Candidates being monitored include:

- **interactive-browser:** runtimes that perform multi-step site navigation, clicking buttons, filling forms, following multi-step flows. Examples emerging in 2026 include Anthropic Computer Use, browser-automation agents. The boundary with `browser-execution` is currently unclear — both execute JavaScript, but `interactive-browser` requires stable selectors and working multi-step flows that `browser-execution` does not.
- **crawler-cached:** runtimes that serve responses from a previously-crawled index rather than fetching live. Examples are mostly fading as live retrieval becomes universal; included as a watching brief.

Promotion of either pattern to a fourth category requires: (a) two or more independently-operated runtimes implementing the pattern as their dominant architecture, (b) measurably different readability requirements compared to existing categories, (c) at least 60 days of observed differentiation.

## Characteristic vocabulary

Beyond category, ARC v0.1 defines specific runtime characteristics. These are the dimensions along which runtimes vary within and across categories.

### js_execution (boolean)

Does the runtime execute JavaScript before delivering content to the model?

- `true` — JavaScript executes; CSR content is visible
- `false` — JavaScript does not execute; only the initial HTML payload is visible
- `unknown` (stored in `capabilities.js_execution_status`) — verification pending; treat as `false` for measurement purposes pending confirmation

### session_model (enum)

How does the runtime handle session state across multiple calls within a single agent task?

- `stateless` — each call is independent; no state persists
- `tool-call-state` — state persists within a single agent task but does not survive a crash
- `durable-execution` — state persists across crashes via checkpointing; long-running agent tasks resume from checkpoint after failure

### sandbox_semantics (enum, stored in `capabilities.sandbox_semantics`)

What is the runtime's sandbox model for code execution and external calls?

- `none` — no sandboxing; calls execute in the runtime's main context
- `process-isolation` — calls execute in isolated processes
- `dynamic-workers` — calls execute in Cloudflare-style Dynamic Workers (or equivalent)
- `vm-based` — calls execute in a VM-level isolation boundary

### sub_agent_isolation (boolean, stored in `capabilities`)

Can sub-agents (child agents spawned by a parent agent) execute in isolation from each other and from the parent?

### checkpoint_recovery (boolean, stored in `capabilities`)

Can the runtime recover from a crash mid-task and resume from a saved checkpoint?

### relevant_outcome_pillar (enum)

Which outcome pillar's sub-score does this runtime contribute to?

- `seo` — search-engine crawlers (Googlebot, Bingbot, regional crawlers). Compatibility findings against these runtimes contribute to the SEO pillar's runtime-compatibility sub-score.
- `aeo` — answer-surface fetchers (Google AIO pipeline, Bing Chat answer surface, deterministic snippet fetchers). Compatibility findings against these runtimes contribute to the AEO pillar's runtime-compatibility sub-score.
- `geo` — AI-engine runtimes (ChatGPT, Perplexity, Claude, Gemini, AI Overview's citation-side runtime). Compatibility findings against these runtimes contribute to the GEO pillar's runtime-compatibility sub-score.
- `multiple` — a runtime that genuinely affects more than one pillar's outcome. The implementation documents the multi-pillar routing rule.

The `relevant_outcome_pillar` characteristic is the routing key by which ARC findings flow into outcome pillar sub-scores. It must be specified for every runtime in the registry.

### Emerging characteristics

Characteristics that emerge after v0.1 publishes are added as JSONB keys in the `capabilities`, `measurements`, or `extensions` columns of an implementation's runtime registry. The path-versioning convention (`agent-runtime-compatibility/v0.1`, `agent-runtime-compatibility/v0.2`, etc.) lets implementations track which version their runtime metadata corresponds to.

When a JSONB key proves stable and widely-referenced across 3–6 months, it may be promoted to the typed characteristic vocabulary in v0.2 of this spec. Implementations that have already adopted the JSONB key continue to work without modification; implementations that haven't yet absorb the typed column via migration.

## Measurement framework

### Compatibility check

A compatibility check evaluates whether a website (or specific page) exhibits a particular property required by a particular runtime. The check produces a finding: `compatible`, `incompatible`, `not-applicable`, or `indeterminate`.

A compatibility check is structurally:

```
check_id: string                    e.g. 'css.client-rendered-primary-content'
display_name: string                e.g. 'Primary content rendered client-side'
applies_to_runtimes: string[]       e.g. ['perplexity-fetch', 'gemini-fetch', 'claude-agent-skills']
```

The same conceptual check may evaluate differently against different runtimes. For example, a check `crawler.allows_runtime_access` (does the website's robots.txt allow the runtime's crawler?) evaluates differently for each runtime because each runtime has a different crawler user-agent.

### Applicability matrix

The applicability matrix is the set of (check, runtime) pairs for which the check is meaningful. It is structurally a join: for any check_id and any runtime_id, the matrix returns `true` (applicable), `false` (not applicable), or `null` (not yet defined).

Implementations of ARC populate their own applicability matrices based on their check catalogues. The v0.1 spec does not prescribe specific applicability decisions because check catalogues vary across implementations. The v0.1 spec does prescribe the *structure* of the applicability matrix and the requirement that implementations populate it before producing per-(check × runtime) findings.

### Per-(check × runtime) finding emission

Compatibility findings are emitted per (check × runtime) pair. A single check evaluating against six runtimes produces up to six findings (fewer if some pairs are non-applicable per the matrix). The findings carry a `runtime_id` field linking back to the runtime registry.

This emission pattern produces customer-facing UI of the form:

> Visible to: Cloudflare Project Think, Google Search-as-Agent-Manager, Claude Agent Skills
> Invisible to: OpenAI Agents SDK, Perplexity Fetch, Gemini Fetch
> Not applicable to: [runtimes where applicability matrix is false]

This breakdown is structurally more informative than a single binary "AI-compatible" verdict because it exposes which categories of runtime fail and why.

### Routing to outcome pillars

Compatibility findings route to the outcome pillar specified by the runtime's `relevant_outcome_pillar` characteristic. The routing rule is:

```
finding.routes_to_pillar = runtime.relevant_outcome_pillar
```

For runtimes with `relevant_outcome_pillar = 'multiple'`, the implementation documents which pillars the finding contributes to and at what weight. The default for `multiple` is even distribution across the listed pillars.

The pillar sub-score for runtime compatibility is the aggregate of compatibility findings routed to that pillar, calculated per the pillar spec.

### Measurement provenance

Every finding records the methodology version it was produced under. Findings produced under ARC v0.1 carry `methodology_ref = 'agent-runtime-compatibility/v0.1'`. When ARC promotes to v0.2 or v1.0, existing findings retain their original methodology_ref; new findings carry the new version.

Findings may carry dual methodology_refs when a single check has overlapping methodologies (e.g. a check measuring AI crawler accessibility surfaces both `ai-crawler-accessibility/v0.1` and `agent-runtime-compatibility/v0.1` because crawler-blocking equates to runtime-blocking for headless-fetch runtime category). The dual-reference syntax is `+`-separated: `ai-crawler-accessibility/v0.1+agent-runtime-compatibility/v0.1`.

## Customer-site readability framework

This section provides the concrete tests a website operator can run to assess their site's runtime readability. The tests map to the three questions Manić's May 2026 framing introduced, expanded with category-specific guidance.

### Test 1 — Machine-readable structured responses

**Question:** Do your most important endpoints return machine-readable structured responses, or do they only render correctly inside a full browser session?

**Method:**

1. Identify the website's "important endpoints" — typically the home page, product/service pages, contact and pricing pages, and any pages targeted by AI engine citations (audit history shows which pages get cited).
2. For each endpoint, fetch the URL with `curl` or equivalent without JavaScript execution. Inspect the HTML payload.
3. Verify primary content is present in the initial HTML: headlines, body text, schema markup, structured data.
4. If primary content is absent or empty (CSR-only rendering), the endpoint is invisible to all runtimes in the `headless-fetch` category and to any `model-native-harness` runtime that defaults to headless fetch.

**Compatibility verdict:**

- Primary content present in initial HTML → compatible with `headless-fetch`, `browser-execution`, `model-native-harness` (typically all categories)
- Primary content rendered only after JavaScript execution → compatible with `browser-execution`; incompatible with `headless-fetch`, `model-native-harness`

### Test 2 — Agent-actor session continuity

**Question:** Is your authentication scoped so an agent acting on a user's behalf can hold a session across multiple calls, or does it only support one-shot human logins?

**Method:**

1. Identify any endpoints that require authentication (logged-in pricing, member-only content, customer dashboards).
2. Verify the session pattern: does authentication produce a token that can be reused across multiple subsequent calls within a single agent task?
3. Verify the session does not break under concurrent calls (typical of `tool-call-state` and `durable-execution` runtimes that may issue parallel sub-agent calls).
4. Verify the session does not impose human-only friction (CAPTCHAs, browser fingerprinting, mouse-movement detection) that an agent cannot reasonably satisfy.

**Compatibility verdict:**

- Session-token reusable across calls → compatible with `tool-call-state`, `durable-execution`
- Session-token tied to a single browser fingerprint → incompatible with most agent runtimes
- No authentication required → not applicable

### Test 3 — Structured data invariance under headless fetch

**Question:** Does your structured data still mean the same thing if a runtime that did not execute your JavaScript tried to read it?

**Method:**

1. Fetch the endpoint without JavaScript execution.
2. Parse the HTML for `application/ld+json` blocks, microdata, RDFa.
3. Verify the structured data semantically matches the rendered page content. Common failure: the structured data describes a product detail page, but the JavaScript-rendered content is a category listing — the runtime sees mismatched signals.
4. Verify the structured data validates against schema.org (or whatever vocabulary it uses).

**Compatibility verdict:**

- Structured data matches HTML content → compatible with all runtime categories
- Structured data missing or invalid → reduced compatibility across all categories
- Structured data injected by JavaScript only → invisible to `headless-fetch` runtimes; visible to `browser-execution` runtimes

### Test 4 — Crawler permission alignment

**Question:** Does your robots.txt and meta robots configuration explicitly grant the runtime's crawler the access required to read your content?

**Method:**

1. Identify the runtime's crawler user-agent. Examples as of May 2026:
   - Cloudflare Project Think: `cf-think/*` (fetch via Worker proxy; not a crawler in the traditional sense)
   - OpenAI Agents SDK: `GPTBot` for general fetches; specific user-agents for SDK calls
   - Google Search-as-Agent-Manager: `Googlebot` (inherits search crawler)
   - Claude Agent Skills: `ClaudeBot`
   - Perplexity Fetch: `PerplexityBot`
   - Gemini Fetch: `Google-Extended` for AI training and inference contexts
2. Verify robots.txt explicitly allows the user-agent for the URLs the agent would fetch.
3. Verify meta robots and X-Robots-Tag headers do not impose `noai`, `noLLM`, or `noimageai` directives that block the runtime.
4. Verify the website does not implement bot-detection systems that block the user-agent.

**Compatibility verdict:**

- All applicable user-agents explicitly allowed → compatible
- User-agent blocked or implicitly disallowed → incompatible for that runtime
- User-agent not addressed in robots.txt → indeterminate (depends on default behaviour)

This test maps to the AI crawler accessibility methodology spec (`ai-crawler-accessibility/v0.1`); findings are dual-referenced.

### Test 5 — Agent-skills declaration

**Question:** Does your site publish a discoverable description of what agent-relevant operations or content categories it supports?

**Method:**

1. Check for the presence of `/.well-known/agent-skills.json` or equivalent agent-skills declaration (the file path convention is emerging in 2026 via tools such as isitagentready.com and Anthropic's agent-skills primitives).
2. If present, verify the file is well-formed JSON, includes the expected fields per the emerging convention, and accurately describes operations the site supports.
3. Note: this test is informational at v0.1. The file format is not yet formalised as a W3C standard, but adoption is observable across early-mover sites. Implementations should record presence/absence as a finding without strong-weighting the absence as failure.

**Compatibility verdict:**

- Well-formed agent-skills declaration present → compatible (informational positive)
- Declaration absent or malformed → indeterminate at v0.1 (no penalty; informational only)

This test will receive a more definitive weighting decision in v0.2 once the file-format convention stabilises.

### Test 6 — Markdown alternates

**Question:** For pages targeted by AI engine citations, are clean Markdown alternates available for runtimes that prefer them?

**Method:**

1. Identify pages that are commonly cited or are candidates for citation (typically: home, product/service pages, key articles).
2. Check whether each page has a `.md` or Markdown alternate available at a predictable URL pattern (e.g. `page.md` adjacent to `page.html`, or `/api/markdown/page` endpoint).
3. Check `llms.txt` for the site (if present) — does it advertise Markdown alternates?
4. Note: like Test 5, this is informational at v0.1. Markdown-alternate availability is observably preferred by some runtimes (Claude, certain RAG pipelines) but the convention is not formalised.

**Compatibility verdict:**

- Markdown alternates available and discoverable → compatible (informational positive)
- Not available → indeterminate at v0.1 (no penalty; informational only)

## Derived Agent Runtime Readiness view

The derived Agent Runtime Readiness view provides a cross-pillar summary of runtime compatibility. It is reported alongside the three pillar scores in customer-facing UI but is not a pillar score and does not contribute to the overall MakeitSEO score.

### Computation

The Agent Runtime Readiness indicator is computed from the three pillars' runtime-compatibility sub-scores:

```
runtime_readiness = (
  seo.runtime_compatibility_subscore +
  aeo.runtime_compatibility_subscore +
  geo.runtime_compatibility_subscore
) / 3
```

This is the arithmetic mean of the three pillars' runtime sub-scores, on a 0–100 scale.

### Display

Customer-facing UI surfaces the readiness indicator alongside the three pillar scores. The display shows:

- The overall readiness score (0–100)
- A breakdown by runtime: which runtimes can read the site, which cannot, which are not applicable
- A breakdown by pillar consequence: where runtime failures are most affecting outcomes

The breakdown is structurally:

| Runtime | Category | Visible | Incompatible | N/A | Affects pillar |
|---|---|---|---|---|---|
| Googlebot | browser-execution | ✓ | | | SEO |
| Bingbot | headless-fetch | ✓ | | | SEO |
| Google AIO pipeline | browser-execution | ✓ | | | AEO |
| ChatGPT runtime | model-native-harness | | ✗ | | GEO |
| Perplexity Fetch | headless-fetch | ✓ | | | GEO |
| Claude Agent Skills | model-native-harness | | ✗ | | GEO |
| Gemini Fetch | headless-fetch | ✓ | | | GEO |

### Tier-gated runtime visibility

The runtime registry is tier-gated per the MakeitSEO pricing tiers. Smaller tiers see fewer runtimes in the readiness breakdown; larger tiers see the full set. Tier visibility is operator-configured in the runtime registry. The structural finding ("visible to X, invisible to Y") is accessible at all tiers; only the breadth of runtimes covered varies.

## Implementation guidance

### Runtime registry structure

Implementations should maintain a runtime registry: a structured catalogue of known runtimes with their characteristics. The registry should support:

- Listing runtimes filtered by active status, tier visibility, category, relevant outcome pillar
- Adding new runtimes as the field evolves
- Toggling runtime active status without code deploys
- Storing emerging characteristics in extensible JSONB-equivalent fields

Pattern 1 (recommended): structured columns for known dimensions (id, name, category, relevant_outcome_pillar, js_execution, session_model) plus JSONB columns for emerging characteristics (capabilities, measurements, extensions), with explicit promotion path when JSONB keys prove stable.

### Applicability matrix structure

Implementations should maintain an applicability matrix: a join table or equivalent structure mapping check_id × runtime_id to is_applicable status with optional weight modifier and notes.

The matrix should be operator-editable through an admin interface. Default applicability rules can be derived from runtime category (e.g. session-continuity checks are not applicable to stateless runtimes by default) but explicit per-cell overrides should be supported.

### Per-(check × runtime) audit pipeline

Implementations producing compatibility findings should emit findings per (check × runtime) pair where the matrix marks the check applicable. The findings table should carry a `runtime_id` field; downstream UI should surface the runtime breakdown to operators and customers. Each finding's pillar routing is determined by the runtime's `relevant_outcome_pillar` characteristic.

### Cascade-monitoring data collection

For implementations participating in the MakeitSEO Standards Programme's cascade-weighting v0.2 review (per `methodology-adr/v0.1` Deferred 2), the audit pipeline must:

- Time-pair per-runtime ARC compatibility findings with per-platform GEO citation outcomes
- Tag customer cohorts by cascade-relevant traits (CSR-heavy, SSR, mixed-architecture, crawler-blocked, fully open)
- Retain findings and outcomes at the granularity needed for correlation analysis

This data feeds the 60-day and 90-day reviews that determine whether cascade weighting is introduced in v0.2.

### Customer-facing surface

Customer-facing UI should present:

- The three pillar scores (SEO, AEO, GEO) composing the overall MakeitSEO score
- The derived Agent Runtime Readiness indicator alongside, with the runtime breakdown one click away
- Per-pillar runtime sub-scores accessible from the pillar detail view

Runtime compatibility should not be hidden as a technical detail. It is a primary dimension of the audit and a primary lever for customer action.

## Edge cases

### Runtime not in registry

When an audit encounters a runtime not in the registry (typically because the runtime is new and the registry hasn't been updated), the audit emits the finding with `runtime_id = NULL` and a "runtime not yet classified" status. Operator review via admin route adds the runtime to the registry with its relevant_outcome_pillar designation.

### Partial JS execution

Some runtimes execute JavaScript with limitations (timeout limits, no third-party network access, blocked browser APIs). These runtimes are categorised as `browser-execution` but the limitations are captured in `capabilities.js_execution_constraints`. Compatibility checks may need to flex to "JavaScript executes but with constraint X" rather than binary "JS executes / doesn't execute."

### Runtime evolution within a category

Runtimes evolve. A runtime that ships in `headless-fetch` category in May 2026 may add JavaScript execution capability by November 2026 and migrate to `browser-execution`. Implementations should update the runtime registry as the field evolves; the methodology_ref versioning ensures findings remain interpretable in the context of the version they were produced under.

### Conflicting signals across runtimes

A website may be compatible with some runtimes and incompatible with others — this is the normal case, not an edge case. Customer-facing UI should present this clearly without forcing a binary verdict. Each finding routes to its pillar's runtime sub-score independently.

### Crawler blocking as runtime blocking

For headless-fetch and model-native-harness runtimes, crawler blocking equates to runtime blocking. A robots.txt that disallows `PerplexityBot` makes the website invisible to Perplexity Fetch entirely. Findings for crawler-blocking checks should be dual-referenced (`ai-crawler-accessibility/v0.1+agent-runtime-compatibility/v0.1`) to surface both the crawler-level and runtime-level implications.

### Runtime affects multiple pillars

Some runtimes legitimately affect more than one pillar's outcome (e.g. Googlebot affects both SEO ranking and AEO inclusion in featured snippets and Google AI Overviews). For these, the `relevant_outcome_pillar` characteristic is set to `multiple` and the implementation documents the per-pillar routing rule. The default is even distribution across the listed pillars.

## Examples

### Example 1: SSR marketing site, schema in initial HTML

A traditional server-side-rendered marketing site with JSON-LD schema markup in the initial HTML payload, compliant robots.txt, and no authentication-gated content.

Compatibility findings:

- vs `cloudflare-project-think` (relevant_outcome_pillar=multiple): compatible → contributes to AEO + GEO
- vs `openai-agents-sdk` (relevant_outcome_pillar=geo): compatible → contributes to GEO
- vs `google-search-agent-manager` (relevant_outcome_pillar=multiple): compatible → contributes to SEO + AEO
- vs `claude-agent-skills` (relevant_outcome_pillar=geo): compatible → contributes to GEO
- vs `perplexity-fetch` (relevant_outcome_pillar=geo): compatible → contributes to GEO
- vs `gemini-fetch` (relevant_outcome_pillar=geo): compatible → contributes to GEO

Pillar runtime sub-scores:

- SEO runtime sub-score: 100 (all SEO-relevant runtimes visible)
- AEO runtime sub-score: 100 (all AEO-relevant runtimes visible)
- GEO runtime sub-score: 100 (all GEO-relevant runtimes visible)

Derived Agent Runtime Readiness: 100.

### Example 2: CSR React SPA, schema injected via useEffect

A React single-page application that renders content client-side, including schema markup injected after page load via a useEffect hook.

Compatibility findings:

- vs `cloudflare-project-think`: compatible (browser-execution; JS runs)
- vs `openai-agents-sdk`: incompatible (model-native-harness defaults to headless; schema invisible)
- vs `google-search-agent-manager`: compatible (Googlebot renders JS)
- vs `claude-agent-skills`: incompatible (model-native-harness; headless fetch)
- vs `perplexity-fetch`: incompatible (headless-fetch; no JS)
- vs `gemini-fetch`: incompatible (headless-fetch; no JS)

Pillar runtime sub-scores (illustrative; actual aggregation per pillar spec):

- SEO runtime sub-score: ~100 (Googlebot renders JS; SEO-relevant runtime is satisfied)
- AEO runtime sub-score: ~50 (Google AIO pipeline can read, but Bing Chat surface may not)
- GEO runtime sub-score: ~25 (only 1 of 4 GEO-relevant runtimes can read the site)

Derived Agent Runtime Readiness: ~58. The CSR architecture creates significant GEO invisibility. Recommendation: server-side render at least the schema markup and primary content in the initial HTML payload to recover compatibility with `headless-fetch` and `model-native-harness` runtimes.

### Example 3: Authenticated SaaS dashboard

A logged-in customer dashboard requiring username + password + 2FA authentication.

Compatibility findings:

- vs `cloudflare-project-think`: indeterminate (durable-execution can hold session, but 2FA blocks initial authentication)
- vs `openai-agents-sdk`: indeterminate (similar — tool-call-state with auth tools possible, but 2FA hard)
- vs all others: not-applicable for the authenticated content; applicable for any public marketing pages

This is an architectural decision for the operator: do you want agents acting on a user's behalf to access this content? If yes, an agent-actor authentication scheme is required. If no, the current state is correct.
