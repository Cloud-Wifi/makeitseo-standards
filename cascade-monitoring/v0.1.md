## Abstract

This specification defines the cascade-monitoring infrastructure that detects meaningful changes in the underlying methodology signals (brand mentions, content freshness, per-platform affinity, structured-content density) and emits severity-graded events for operator and customer attention.

Cascade monitoring is an operational layer rather than a scoring layer. It does not contribute to any pillar score. Its purpose is twofold:

1. **Customer-facing alerting:** when a customer's underlying signals drift materially, surface the change as a finding before the next audit cycle re-computes the pillar score
2. **v0.2 cascade-weighting evidence:** capture the time-series data needed to evaluate whether ARC runtime failures should additionally depress citation outcomes in the GEO pillar (per `methodology-adr/v0.1` Deferred 2)

The spec defines four event types at v0.1, three severity levels, and the data structure for the events. Per-event-type detection thresholds are documented; v0.2 may refine the thresholds against cohort data.

---

## Status

This is the v0.1 draft of cascade monitoring. The implementation captures event data from launch onward; the principal v0.2 review (cascade-weighting decision at 60–90 days post-launch) depends on this data being collected continuously.

The spec is intentionally minimal. The principal methodological choices are: (a) which signals to monitor, (b) what thresholds trigger events, (c) what severity is assigned. All are documented here.

## Scope

### In scope

- The four event types at v0.1: `brand_mention_spike`, `freshness_drop`, `affinity_shift`, `schema_gap`
- Per-event-type detection thresholds
- Severity-grading rules
- Acknowledgment workflow for operator-side event handling
- Edge cases identified at v0.1
- v0.2 cascade-weighting evidence collection requirements

### Out of scope

- Direct customer-facing alerts (the events are operator-side at v0.1; customer-facing alerting via the events is a UX choice for v0.2)
- Real-time event streaming (v0.1 events are produced per measurement cycle, not continuously)
- Cross-customer cohort analysis (operator-side analyses run against the events table; v0.1 doesn't formalise the analysis methodology)
- Automated remediation (events surface problems; humans decide what to do)

## Definitions

### Cascade event

An emitted record indicating that a measured signal has changed beyond a threshold within a measurement window. Events carry a type, severity, payload, and acknowledgement state.

### Event type

A classification of the kind of change detected. v0.1 defines four event types: `brand_mention_spike`, `freshness_drop`, `affinity_shift`, `schema_gap`.

### Severity

A classification of the urgency of operator attention required. Three levels: `info` (informational; no operator action expected), `warning` (operator review recommended), `critical` (operator action required).

### Acknowledgement

The operator's marking of an event as reviewed. An acknowledged event remains in the database for forensics but is removed from active alert queues.

### Cascade-weighting

The methodological question (from `methodology-adr/v0.1` Deferred 2) of whether ARC runtime failures should additionally depress citation outcomes in the GEO pillar beyond the runtime sub-score effect. The v0.2 review of this question depends on event data collected via cascade monitoring.

## Event types at v0.1

### brand_mention_spike

**What it detects:** A material increase in the customer's brand web mention volume within the prior 7-day window compared to the 30-day prior baseline.

**Threshold:** mention volume in the prior 7 days ≥ 2× the mean weekly volume across the 30 days preceding the spike window.

**Severity assignment:**
- `info` if the spike is 2–3× baseline (could be normal seasonal or PR variation)
- `warning` if 3–5× baseline (notable; worth operator attention)
- `critical` if 5×+ baseline (extreme; investigate for PR event, paid placement, or measurement-anomaly)

**Why monitor it:** A brand mention spike usually has a cause — press coverage, viral content, paid placement campaign, or measurement anomaly (DataForSEO index update, brand-collision noise). Each cause has different score implications and customer-facing communication.

### freshness_drop

**What it detects:** A material decrease in the customer's domain-level content freshness score between consecutive audit cycles.

**Threshold:** freshness score drops by ≥ 15 points between the current audit and the most recent prior audit within the prior 60 days.

**Severity assignment:**
- `info` if the drop is 15–25 points (publishing slowdown or natural ageing of recent content)
- `warning` if 25–40 points (notable shift; the customer's content-update cadence has changed)
- `critical` if 40+ points (extreme; possible CMS issue or strategic publishing change requiring operator review)

**Why monitor it:** Freshness drops can indicate publishing slowdowns (a customer reducing content investment), technical issues (CMS not updating `dateModified` correctly), or natural ageing (no new content added; existing content cycling through age bands).

### affinity_shift

**What it detects:** A material change in the customer's per-platform affinity score on any single platform between consecutive measurement windows.

**Threshold:** relative affinity on a platform changes by ≥ 0.5 (i.e. moves from 1.0× to 1.5× or from 1.5× to 1.0×, etc.) between consecutive 30-day windows.

**Severity assignment:**
- `info` if one platform shifts by 0.5–0.75 (single-platform variability; can be noise)
- `warning` if one platform shifts by 0.75–1.25 OR two platforms shift simultaneously (more notable)
- `critical` if one platform shifts by 1.25+ OR three+ platforms shift simultaneously (extreme; possible platform-algorithm change or customer-side change in fortunes)

**Why monitor it:** Platform-specific affinity shifts often reveal algorithm-level changes in the engine. A 0.5+ shift in ChatGPT affinity for a customer with otherwise stable signals may reflect a ChatGPT retrieval-system update. Aggregated across customers, these shifts inform the v0.2 cascade-weighting review.

### schema_gap

**What it detects:** A material reduction in the customer's structured-content (schema markup) coverage between consecutive audits.

**Threshold:** the count of distinct schema types detected across tracked pages drops by ≥ 30% between consecutive audits, OR the count of pages with structured content drops by ≥ 25%.

**Severity assignment:**
- `info` if drop is 25–40%
- `warning` if 40–60%
- `critical` if 60%+ (extreme; possible site-wide template change or schema-injection failure)

**Why monitor it:** Schema gaps usually indicate technical issues — a CMS template change that dropped schema markup, a deployment that broke a structured-data injection, or a content migration that lost markup. Detecting this early prevents the schema-coverage-driven sub-score from silently degrading.

## Severity-grading rules

Severity is assigned per the per-event-type thresholds above. The implementation may additionally apply cross-event severity escalation:

- If two or more `warning` events fire for the same customer within a 7-day window, the latter event is escalated to `critical`
- If a `critical` event fires within 14 days of a previous `critical` event for the same customer (same or different event type), the latter event is annotated as a "compound critical" — same severity but flagged in the operator UI as part of a sequence

These escalation rules are deliberately conservative at v0.1. v0.2 may introduce more aggressive cross-event correlation (e.g. "a brand_mention_spike followed by an affinity_shift in the same direction within 14 days is suggestive of a successful PR cycle producing AI citation").

## Acknowledgement workflow

Events default to `acknowledged_at = NULL` (un-reviewed). An operator marks an event acknowledged via the admin route; this sets `acknowledged_at = NOW()` and removes the event from active alert queues.

Acknowledgement does not delete the event. The event remains in the database for forensics. Operators may add an acknowledgement note (free-text payload field) describing the diagnosis or action taken.

Auto-acknowledgement is not done at v0.1. Every event requires explicit human review to leave the active queue. v0.2 may introduce auto-acknowledgement for `info`-severity events older than 14 days, but v0.1 keeps the workflow strictly operator-driven.

## Detection procedure

### Per-event-type detection cadence

Cascade events are detected at the end of each measurement cycle for the underlying signal:

- `brand_mention_spike`: detected weekly (after the weekly brand-mention cron runs)
- `freshness_drop`: detected per audit (after each audit's freshness measurement)
- `affinity_shift`: detected weekly (after the weekly platform-affinity cron runs)
- `schema_gap`: detected per audit (after each audit's structured-content measurement)

The detection step is part of the same cron or audit pipeline that produces the underlying measurement. No separate cron runs; events emerge as a side effect of routine measurement work.

### Methodology version tagging

Every cascade event carries `methodology_ref = 'cascade-monitoring/v0.1'`. Events may carry secondary references to the spec governing the underlying signal (e.g. a brand_mention_spike event carries `'cascade-monitoring/v0.1+brand-web-mentions/v0.1'`).

### Storage

Events are stored in `cascade_monitoring_events` with fields per the Sprint 2.14 migration:
- `id`, `domain_id`, `event_type`, `detected_at`, `severity`, `payload` (JSONB), `acknowledged_at`, `methodology_ref`

The `payload` field captures event-specific context (the threshold values, the measurement window dates, the before/after numbers). This is the principal forensic record.

## v0.2 cascade-weighting evidence

Per `methodology-adr/v0.1` Deferred 2, the cascade-monitoring data feeds a 60-day and 90-day review of whether ARC runtime failures should additionally depress citation outcomes in the GEO pillar.

The decision criteria documented in `methodology-adr/v0.1`:
- Pearson r > 0.5 between ARC failures and citation drops across cohorts → introduce cascade weighting in v0.2
- r < 0.3 → retain status quo; document negative finding
- 0.3 ≤ r ≤ 0.5 → defer further to v0.3 with continued data collection

The relevant data for this analysis is captured in:
- `cascade_monitoring_events` rows of type `affinity_shift` (citation outcome changes)
- `arc_pillar_findings` rows tagged for the GEO pillar (ARC runtime findings routed to GEO)
- Time-paired across the customer's audit history

The implementation provides an admin route `GET /admin/cascade-monitoring/cascade-correlation-analysis` that runs the Pearson correlation across cohort-tagged data for a configurable time window and cohort filter. The analysis is operator-initiated at the 60-day and 90-day review milestones.

### Cohort tagging

For cascade-weighting analysis, customers should be tagged by `runtime_compatibility_cohort` indicating their structural profile:

- `csr_heavy`: > 80% of pages CSR-rendered
- `ssr_clean`: > 80% of pages SSR-rendered AND all crawlers explicitly allowed
- `mixed`: 20-80% CSR ratio
- `crawler_blocked`: robots.txt blocks any GEO-relevant crawler
- `fully_open`: ssr_clean AND all crawlers allowed
- `undetermined`: insufficient signal to classify

Cohort tagging is derived from audit findings rather than stored explicitly per customer. The admin route computes cohort membership at analysis time.

Note: the Sprint 2.14 implementation did not include a separate cohort-classifier service or a cohort-tag column. Cohort tagging is computed at analysis time from existing audit data; this matches the principle of "compute on read where possible" rather than storing derived data. v0.2 may introduce stored cohort tags if the analysis becomes frequent enough to justify pre-computation.

## Implementation guidance

### Admin route for event inspection

The implementation provides admin routes for event management:

- `GET /admin/cascade-monitoring/events?status=unacknowledged&severity=critical` — list events for review
- `POST /admin/cascade-monitoring/events/:id/acknowledge` — mark event acknowledged with optional note
- `GET /admin/cascade-monitoring/events/:id` — full event detail including payload

These are admin-only; customer-facing visibility of events is a v0.2 UX decision.

### Event-to-customer communication

v0.1 does not automatically communicate events to customers. The operator reviews events and decides whether to surface them. Common operator actions:

- `info` events: usually no customer communication; logged for trend analysis
- `warning` events: typically surfaced in the next customer audit summary as a "noteworthy change" finding
- `critical` events: typically prompt direct outreach (email, in-app message) to the customer

v0.2 may automate some of this with operator-configurable rules.

### Notification fan-out

For `critical`-severity events, the implementation should emit a notification to the operator team via the existing operator-notification channel (email, Slack, or whatever is configured). v0.1 sends an email to the operator's primary admin address for each `critical` event; v0.2 may introduce per-event-type routing and per-operator-team configurability.

## Edge cases

### Customer's data is too sparse to detect anomalies

A customer with thin data (very few measurements, very new domain) may not trigger any events because the baselines required for threshold computation aren't available. v0.1 silently skips event detection in these cases; no events are emitted, no operator action is required.

The customer-facing finding may note "cascade monitoring requires N measurements; this customer is currently at K/N" if the operator wants to surface this.

### A genuine market event causes simultaneous spikes across many customers

A major market event (e.g. a regulatory change affecting an industry, a viral story affecting many brands) may cause simultaneous brand_mention_spike events across many customers. v0.1 does not detect cross-customer correlation; each customer's event is emitted independently.

Operators should expect occasional event clustering at the cohort level and should not interpret a coincident series of warning events as critical without per-event context.

### Threshold-crossing during a known operator-side change

When the operator deliberately changes the underlying measurement methodology (e.g. expanding the tier-1 brand-mention allowlist), the resulting changes in customer measurements may trip cascade events. Operator should annotate the changes in the operator log; this prevents misinterpretation of the resulting events as customer-side changes.

### Acknowledgement with no remediation

An operator may acknowledge an event with a note like "investigated; no remediation needed; expected variation." The note is preserved in the acknowledgement payload. The event remains a part of the cohort-analysis data; acknowledgement does not exclude the event from the cascade-weighting analysis.

### Customer with `acknowledged_at = NULL` events older than 90 days

Stale unacknowledged events accumulate if the operator does not review consistently. v0.1 does not auto-purge or auto-acknowledge; the operator decides. v0.2 may introduce a "stale event report" that summarises unacknowledged events older than 90 days.

## Examples

### Example 1: B2B SaaS with PR-driven brand mention spike

A B2B SaaS gets coverage in a major tech publication after a product launch announcement.

- Brand mention volume in the prior 7 days: 450 quality-weighted mentions
- 30-day prior baseline weekly mean: 92 quality-weighted mentions
- Ratio: 450 / 92 = 4.9×

Event emitted:
- `event_type = 'brand_mention_spike'`
- `severity = 'warning'` (3–5× baseline)
- `payload = { window: '7d', current: 450, baseline_weekly_mean: 92, ratio: 4.9, top_sources: [TechCrunch, ProductHunt, ...] }`

Operator reviews, recognises the PR cycle as the cause, acknowledges with note "ProductHunt + TechCrunch coverage following X launch."

### Example 2: Customer's CMS template change drops schema markup

A customer rolls out a new theme on their CMS without realising the new theme dropped JSON-LD schema across the site.

- Previous audit: 8 distinct schema types detected across 45 pages with structured content
- Current audit: 2 distinct schema types detected across 12 pages with structured content
- Drop in distinct types: 75%; drop in structured-content pages: 73%

Event emitted:
- `event_type = 'schema_gap'`
- `severity = 'critical'` (60%+ drop)
- `payload = { previous_types: 8, current_types: 2, previous_pages: 45, current_pages: 12, drop_pct_types: 0.75, drop_pct_pages: 0.73 }`

Operator inspects, identifies the template change, contacts the customer with remediation guidance: "Your recent theme update appears to have removed JSON-LD schema. We recommend re-injecting Article/Organization/BreadcrumbList schema. This change is degrading your GEO sub-score 6 (multi-schema + structured content)."

### Example 3: Cross-platform affinity shift

A customer's relative affinity on ChatGPT shifts from 1.20× to 0.55× in a single 30-day window. No corresponding shifts on other platforms.

Event emitted:
- `event_type = 'affinity_shift'`
- `severity = 'warning'` (single-platform shift of ~0.65; just under the 0.75 threshold for warning, but escalated because the magnitude is large)
- `payload = { platform: 'chatgpt', previous_relative: 1.20, current_relative: 0.55, delta: -0.65 }`

Operator inspects, considers possible causes:
1. ChatGPT retrieval-system update (cross-customer affinity shifts would confirm)
2. Customer-side change (e.g. customer rebranded; brand variants in measurement need refresh)
3. Measurement-cohort change (the platform_avg_affinity baseline shifted; check)

Operator runs cross-customer query: if 3+ other customers show similar ChatGPT affinity drops in the same window, the cause is the engine, not customer-side. If isolated, the cause is customer-side.
