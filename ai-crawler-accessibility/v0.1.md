## Abstract

This specification defines how AI crawler accessibility is measured for use as an input to scoring pillars in the MakeitSEO Standards Programme. AI crawler accessibility is the property of a website explicitly granting AI engine crawlers the access they need to fetch, index, and reason over the site's content.

Accessibility is consumed as a sub-score by one pillar: `geo-scoring/v0.1` at 10% weight. Additionally, AI crawler accessibility contributes to the cross-cutting Retrievability derived view (see `methodology-adr/v0.1` Decision 15 and `methodology-overview/v0.1`).

The signal is binary in effect (a blocked crawler cannot cite the site) but is measured across a set of distinct AI crawlers, each evaluated independently. Aggregation produces a 0–100 sub-score reflecting the breadth of AI crawler permission.

---

## Status

This is the v0.1 draft of AI crawler accessibility measurement. The user-agent list is current as of May 2026 and is expected to evolve faster than other parts of the methodology (new AI engines launch; existing engines rename or split their crawlers). v0.2 expects to be a routine refresh; v1.0 expects multiple refreshes.

The spec is intentionally minimal. The principal methodological choices are: (a) which user-agents to measure, (b) what counts as "explicit permission" given that robots.txt semantics permit implicit allowance. Both are documented here.

## Scope

### In scope

- The AI crawler user-agent list at v0.1
- Detection methodology for robots.txt rules, meta robots tags, X-Robots-Tag headers
- Per-crawler accessibility verdict semantics
- Aggregation rule for per-crawler verdicts into a 0–100 sub-score
- Edge cases identified at v0.1

### Out of scope

- Search engine crawler accessibility (Googlebot, Bingbot; these are SEO concerns and would be captured in a separate spec if needed)
- Actual fetch evidence (whether crawlers are observed visiting the site; this is the v0.2 retrievability extension per `methodology-adr/v0.1` Decision 15)
- IP-block-list inspection (operator-side bot management at the firewall layer; not measurable from outside the site)
- Subdomain-specific accessibility (treated at site-root level only at v0.1; per-subdomain crawler permission is a v0.2 candidate)

## Definitions

### AI crawler

A web crawler operated by an AI engine vendor (or its delegated infrastructure provider) for the purpose of fetching site content into the AI engine's retrieval index, training set, or live-retrieval pipeline. AI crawlers identify themselves via the standard `User-Agent` HTTP request header.

### Accessibility verdict

The result of evaluating a single AI crawler's permission to fetch a site. v0.1 defines four verdicts: `explicitly_allowed`, `explicitly_disallowed`, `default_indeterminate`, `noai_directive`.

### robots.txt

The conventional `/robots.txt` file at the site root, governed by the Robots Exclusion Protocol (RFC 9309 as of 2022).

### Meta robots tag

An HTML meta tag (`<meta name="robots" content="...">`) in the page `<head>` providing per-page crawler directives.

### X-Robots-Tag

An HTTP response header (`X-Robots-Tag: ...`) providing crawler directives at the response level, with the same semantics as meta robots but applicable to non-HTML resources.

### noai / noLLM / noimageai directives

Newer directives (emerging across 2024–2026) that specifically restrict AI training and inference use of content. The `noai` directive is the broadest; `noLLM` is more specific to language models; `noimageai` restricts AI use of images. Specifications are vendor-published rather than formal standards as of May 2026.

## AI crawler user-agent list at v0.1

The implementation evaluates accessibility for each crawler in the following list. The list reflects the AI engine crawlers commercially significant as of May 2026.

| Crawler | User-Agent | Purpose | Vendor |
|---|---|---|---|
| GPTBot | `GPTBot` | Training and inference fetches | OpenAI |
| ChatGPT-User | `ChatGPT-User` | User-initiated inference fetches | OpenAI |
| PerplexityBot | `PerplexityBot` | Live retrieval for Perplexity | Perplexity |
| ClaudeBot | `ClaudeBot` | Training and inference fetches | Anthropic |
| Claude-User | `Claude-User` | User-initiated inference fetches | Anthropic |
| Google-Extended | `Google-Extended` | AI training and inference (distinct from Googlebot for ranking) | Google |
| cf-think | `cf-think` (or current ID) | Cloudflare Project Think runtime fetches | Cloudflare |
| CCBot | `CCBot` | Common Crawl indexing (feeds multiple AI training corpora) | Common Crawl Foundation |
| Bytespider | `Bytespider` | TikTok / ByteDance AI fetches | ByteDance |
| Diffbot | `Diffbot` | Diffbot knowledge graph (consumed by multiple AI engines) | Diffbot |
| Applebot-Extended | `Applebot-Extended` | Apple Intelligence AI training | Apple |

The list is operator-maintained. Implementations may add additional crawlers as they emerge. The v0.2 review window is expected to add 2–4 new crawlers based on the AI engine launches in the months following v0.1.

### Crawler tier weighting

For the purposes of the v0.1 sub-score, all crawlers in the list are weighted equally. Each crawler contributes a 1/N share where N is the size of the list. This is a deliberate v0.1 simplification:

- Per-crawler commercial-impact weighting (e.g. GPTBot more important than Bytespider for English-language markets) would require defending specific weight assignments
- The 2026 evidence base does not yet support confident per-crawler weighting
- Equal weighting is honest about what we don't yet know

v0.2 may introduce per-crawler weighting if cohort data shows asymmetric impact on citation outcomes.

## Verdict semantics

For each crawler in the user-agent list, the implementation produces one of four verdicts:

### explicitly_allowed

robots.txt contains a directive specifically permitting the crawler. Examples:

```
User-agent: GPTBot
Allow: /
```

```
User-agent: *
Allow: /
```

with no overriding `User-agent: GPTBot` block.

Or robots.txt is absent and the site responds normally to crawler requests. (Robots Exclusion Protocol semantics treat the absence of robots.txt as full permission.)

### explicitly_disallowed

robots.txt contains a directive specifically denying the crawler. Examples:

```
User-agent: GPTBot
Disallow: /
```

```
User-agent: GPTBot
Disallow: /private
Disallow: /admin
```

with `Disallow: /` covering the principal site content.

### default_indeterminate

robots.txt exists but does not address the specific crawler, and the catch-all `User-agent: *` rule is ambiguous or restrictive in a way that may or may not apply to the AI crawler.

This verdict captures the legally-ambiguous case where the site has not explicitly granted or denied the AI crawler. Most major AI engines treat default-indeterminate as permission (some explicitly state this in their robots policies); v0.1 conservatively flags this as a partial-credit case in scoring (see Aggregation Rule below).

### noai_directive

The site emits `noai`, `noLLM`, or `noimageai` directives via meta robots tags or X-Robots-Tag headers, indicating an explicit AI-restriction independent of robots.txt. Examples:

```html
<meta name="robots" content="noai, noimageai">
```

```http
X-Robots-Tag: noai
```

Most AI crawlers honour `noai`-family directives; this verdict captures the explicit AI-restriction case. The directive is broader than disallow because it applies even to crawlers not specifically named in robots.txt.

### Verdict precedence

When multiple signals apply to a single crawler, precedence is:

1. `noai_directive` (highest precedence; AI-specific directive overrides general rules)
2. `explicitly_disallowed`
3. `explicitly_allowed`
4. `default_indeterminate` (lowest precedence; fallback when no explicit signal)

## Sub-score aggregation

The per-crawler accessibility sub-score is computed as:

```
crawler_accessibility_score = (
  count(explicitly_allowed) × 1.0
+ count(default_indeterminate) × 0.5
+ count(explicitly_disallowed) × 0.0
+ count(noai_directive) × 0.0
) / N × 100
```

Where N is the total crawler list size.

The `default_indeterminate` partial credit reflects the genuine ambiguity: most major engines treat it as permission, but the customer has not made an explicit choice. The 0.5 weighting honours both the typical interpretation and the absence of explicit permission.

`noai_directive` produces 0.0 even though the directive is the customer's explicit decision — the operational reality is that pages with `noai` set are not cited by AI engines that honour the directive. The customer-facing finding should explain that the low score reflects an explicit choice the customer (or their team) made, not a configuration error.

## Measurement procedure

### Robots.txt retrieval

The implementation fetches `https://{customer_domain}/robots.txt` at the start of each measurement cycle. The response is parsed using a standard robots-parser library (the existing platform integration is sufficient).

If robots.txt is unavailable (HTTP 404, network error, timeout) the implementation treats every crawler as `explicitly_allowed` (consistent with RFC 9309 semantics) and emits a finding noting robots.txt is absent. This is not a score reduction; it's an informational note.

### Meta robots inspection

For each tracked page (per the audit pipeline's existing page enumeration), the implementation parses the `<head>` for `<meta name="robots" content="...">` and inspects the content attribute for `noai`, `noLLM`, `noimageai` directives.

If any tracked page emits a `noai`-family directive, the crawler verdicts for that page are updated. The site-level sub-score reflects the most restrictive verdict across the tracked-page set per crawler.

### X-Robots-Tag inspection

The implementation captures HTTP response headers during the audit's page-fetching pass and inspects `X-Robots-Tag` values for `noai`-family directives. Same semantics as meta robots tags; the header takes effect at the response level for any resource type.

### Cadence

AI crawler accessibility runs as part of each audit cycle. The signal changes only when the customer modifies their robots.txt, meta tags, or X-Robots-Tag headers; intermediate refreshes are not needed.

### Methodology version tagging

Every accessibility measurement carries `methodology_ref = 'ai-crawler-accessibility/v0.1'`. Findings consumed by GEO pillar scoring may carry dual references (`ai-crawler-accessibility/v0.1+geo-scoring/v0.1`).

## Implementation guidance

### User-agent list maintenance

The user-agent list is operator-maintained. v0.1 ships with the eleven crawlers listed above. Operator review is recommended quarterly (or when a new AI engine launches commercially). New entries should include:

- Canonical user-agent string
- Vendor and purpose
- Date added to the list
- Any vendor-published policy on robots.txt interpretation (some engines treat absence-of-directive as permission; others require explicit allow)

### Operator-defined exclusions

Some customers may want to exclude specific crawlers from the sub-score calculation (e.g. a publisher who deliberately blocks GPTBot and accepts the score consequence). v0.1 does not provide a UI for this; the operator can flag exclusions at the implementation level.

The customer-facing finding for an excluded-by-choice crawler should clearly note "your robots.txt blocks GPTBot. This is a customer choice; the score reflects the operational reality."

### Reverse-DNS verification

This spec does not require the implementation to perform reverse-DNS verification of crawler IPs (the standard anti-spoofing measure for crawler analysis). v0.1 trusts the user-agent string declared in robots.txt; verification of actual crawler identity at fetch time is a v0.2 candidate alongside the retrievability fetch-evidence extension.

## Edge cases

### Robots.txt blocks all crawlers

A site with `User-agent: *` / `Disallow: /` in robots.txt blocks all crawlers (including search engines and AI engines). The sub-score is 0; the customer-facing finding explains that this affects both SEO and GEO outcomes and may be unintentional. v0.1 does not auto-recommend a fix; the finding flags the issue.

### Robots.txt has syntax errors

A malformed robots.txt may produce undefined-behaviour across different crawlers. v0.1 treats syntax errors as a `default_indeterminate` verdict for affected crawlers and emits a finding ("robots.txt syntax error detected"). The customer is advised to validate via Google Search Console's robots.txt tester or equivalent.

### Robots.txt redirects

A robots.txt that returns a 301/302 redirect to a different URL is followed once. Multi-hop redirects are followed up to three hops; beyond that, the implementation treats the chain as unresolvable and emits a `default_indeterminate` verdict.

### `noai` directive on some pages but not others

A site where some pages emit `noai` and others do not produces mixed verdicts. The site-level sub-score reflects the weighted aggregate across tracked pages: if 30% of tracked pages emit `noai` and 70% do not, the customer's crawler verdict for the affected crawler is partially restricted, and the sub-score is reduced proportionally.

This is computed as:

```
effective_score_for_crawler = (
  pages_with_explicit_allow / total_tracked_pages × 1.0
+ pages_with_indeterminate / total_tracked_pages × 0.5
+ pages_with_disallowed_or_noai / total_tracked_pages × 0.0
)
```

The per-crawler effective scores are then aggregated across the crawler list.

### Crawler not in the v0.1 list

If the customer's robots.txt addresses a crawler not in the v0.1 list (e.g. a niche AI engine, a regional crawler), the implementation logs the unrecognised crawler in a finding ("crawler `XYZBot` detected in robots.txt; not yet in v0.1 measurement scope") but does not adjust the sub-score. The operator may add the crawler to the list at the next maintenance pass.

### Crawl-delay directives

The Crawl-delay directive (`Crawl-delay: 30`) restricts crawl rate but does not block access. v0.1 treats crawl-delay as compatible with `explicitly_allowed` for sub-score purposes. The finding may note "Crawl-delay applied to GPTBot; may reduce effective fetch frequency" but does not reduce the score.

### Honouring vs not-honouring crawlers

Some AI vendors have stated that their crawlers honour robots.txt; others have stated they do not (or have been observed not to). v0.1 does not encode honour-status. The sub-score reflects what the customer has declared, not whether each vendor actually obeys it.

The customer-facing finding may surface honour-status as informational context where relevant (e.g. "Common Crawl honours robots.txt; if blocked, expect reduced AI training set inclusion") but the score itself reflects only the customer's declared policy.

## Examples

### Example 1: Open robots.txt, no AI restrictions

A typical small-business website with permissive robots.txt:

```
User-agent: *
Allow: /
```

No `noai` directives anywhere.

Per-crawler verdicts: all 11 crawlers receive `explicitly_allowed`.

Sub-score: `11 × 1.0 / 11 × 100 = 100`

### Example 2: Explicit GPTBot block

A publisher deliberately blocking OpenAI's training crawler while allowing other AI engines:

```
User-agent: GPTBot
Disallow: /

User-agent: ChatGPT-User
Disallow: /

User-agent: *
Allow: /
```

Per-crawler verdicts: GPTBot and ChatGPT-User are `explicitly_disallowed`. All others (9) are `explicitly_allowed`.

Sub-score: `(9 × 1.0 + 2 × 0.0) / 11 × 100 = 81.8`

### Example 3: Default indeterminate for all AI crawlers

A site with old robots.txt addressing only search engines:

```
User-agent: Googlebot
Allow: /

User-agent: Bingbot
Allow: /

# No AI crawlers addressed
```

Per-crawler verdicts: all 11 AI crawlers receive `default_indeterminate` (the catch-all is not present; AI crawlers are not specifically addressed).

Sub-score: `11 × 0.5 / 11 × 100 = 50`

The customer-facing finding explains: "Your robots.txt does not explicitly address AI crawlers. While most crawlers will treat the absence as permission, explicit `User-agent: *` allowance ensures consistent behaviour."

### Example 4: Site-wide noai directive

A publisher emitting `noai` site-wide via X-Robots-Tag header:

```
X-Robots-Tag: noai, noimageai
```

(emitted on all responses)

Per-crawler verdicts: all 11 crawlers receive `noai_directive` (the precedence rule applies; `noai` overrides robots.txt and meta tags).

Sub-score: `11 × 0.0 / 11 × 100 = 0`

The customer-facing finding explains: "Your site emits `noai` site-wide. This is an explicit customer choice and may be intentional (e.g. paywall protection). The AI crawler accessibility sub-score reflects this choice. If unintentional, the directive can be removed via HTTP server configuration."
